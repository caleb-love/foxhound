Friction Point 1

Model-config audit started in the wrong mode

Initial try

What was tried: the agent assessed whether gpt-5.4 metadata was “right” by heuristic judgment: “looks suspicious,” “looks guessed,” “pricing may be placeholder.”

Why it didn’t work: config correctness here is not inferential. It needed either official docs or explicit local contract documentation. The initial answer created uncertainty instead of a reliable answer.

Correct approach: read the exact config files first, state only what is locally true, then explicitly separate “local config truth” from “provider-doc truth.” If provider truth is needed, require documentation lookup instead of guessing.

Repo prevention

Add to repo/config:

  • Add a ~/.pi/agent/AGENTS.md rule: “For provider/model metadata, do not infer validity from naming. Report local config truth first; use docs lookup for provider truth.”
  • Add a small JSON comment-free companion doc like ~/.pi/agent/MODEL_REGISTRY.md defining which fields are verified vs placeholder.
  • Add a helper script to validate configured models against a known source.

Estimated savings: 4–6 minutes per config audit.

Friction Point 2

Sensitive config inspection leaked the secret before safeguards kicked in

Initial try

What was tried: directly reading ~/.pi/agent/models.json to inspect the model/provider config.

Why it didn’t work: the file contains apiKey. Raw read returned the full secret in the transcript. The agent then had to respond with a rotation warning after the damage was already done.

Correct approach: treat models.json as conditionally sensitive. Use a redaction-first workflow: inspect exact keys without printing secret values, or read then summarize only non-secret fields. If debugging auth, mask by default.

Repo prevention

Add to repo/config:

  • Update ~/.pi/agent/AGENTS.md to classify ~/.pi/agent/models.json as sensitive when apiKey is inline.
  • Add a helper script like ~/.pi/agent/bin/pi-models-redacted that prints provider/model metadata while masking secrets.
  • Document a preferred env-var/file-secret pattern for Pi auth so inline keys are not normal.

Estimated savings: 8–12 minutes plus avoids secret-rotation overhead and risk.

Friction Point 3

Type fixes were patched to the wrong contract first

Initial try

What was tried: in apps/api/src/routes/annotations.test.ts, the session added imports for TraceRecord and ScoreRecord from @foxhound/db and cast test data to those types.

Why it didn’t work: those exports do not exist from @foxhound/db. The change fixed one lint concern but introduced a typecheck failure that surfaced only during the later staged verification pass.

Correct approach: inspect the actual public exports of @foxhound/db before importing types, or avoid fake-public-type imports in tests and use repository-approved test typing patterns.

Repo prevention

Add to repo/docs:

  • Add a short testing note in CLAUDE.md or docs/reference/engineering-notes.md: “Before importing DB record types in tests, verify they are exported from @foxhound/db; prefer Awaited<ReturnType<...>> or as never only as last-resort local test shims.”
  • Add package export guidance in packages/db/src/index.ts docs or README.
  • Add a narrow preflight script for changed-package typecheck before broader staging.

Estimated savings: 6–10 minutes per similar test-fix cycle.

Before: fix that later failed typecheck
