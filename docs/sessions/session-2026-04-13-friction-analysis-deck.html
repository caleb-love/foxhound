Friction point 3 · Tried unsupported Fastify route introspection

Initial wrong direction

What was tried

Attempted a route-config regression test using direct route imports plus app.routes[...] inspection.

write apps/api/src/routes/rate-limit-config.test.ts → imports route modules directly → expects app.routes to exist pnpm --filter @foxhound/api test -- src/routes/rate-limit-config.test.ts → DATABASE_URL environment variable is required pnpm --filter @foxhound/api typecheck → Property 'routes' does not exist on FastifyInstance
Correct direction

What worked

Backed out the bad test immediately and returned to behavior-based verification instead of unsupported config introspection.

rm apps/api/src/routes/rate-limit-config.test.ts pnpm --filter @foxhound/api typecheck → green again

Why it didn’t work

Route modules pull real dependencies in this repo. Direct import hit live DB client requirements. Also, the assumed Fastify route registry API was wrong.

Repo improvement to prevent it

  • Add a testing note to docs/reference/engineering-notes.md:
  • “Do not use route-config introspection via direct route imports in API tests; prefer behavior tests or a documented app-factory helper with mocks.”
  • Add a dedicated mocked app-factory helper if config assertions are desired later.

Estimated time savings: 7–12 minutes/session.

Friction point 4 · Initially aimed at lower-ROI frontend tests

Initial wrong direction

What was tried

Started thinking in terms of server-page unhappy-path tests for traces/detail/diff.

inspect server page files consider page-level tests around getServerSession / redirect / fetch failure later realization: real flakiness is in TraceTable and interactive client state
Correct direction

What worked

Pivoted to reusable page-state shells plus high-ROI client interaction tests.

write apps/web/components/ui/page-state.tsx edit traces/detail/diff pages to use shared state shells write login/page.test.tsx write trace-table.test.tsx write trace-timeline.test.tsx write run-diff-view.test.tsx

Why it didn’t work

The buggy surface was mostly client interaction: compare selection, loading states, transient failures, and click behavior. Server-page tests would have been more brittle and less representative.

Repo improvement to prevent it

  • Add a frontend testing rule to repo guidance:
  • “For wireframe UI, prioritize client interaction components over server-page shells.”
  • Document reusable page-state components as the default for empty/error/warning shells.

Estimated time savings: 4–8 minutes/session, plus higher test ROI.
