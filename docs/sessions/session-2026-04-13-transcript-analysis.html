Friction Point 2

Ran broader typecheck expecting prompt-slice verification, but hit unrelated fixture debt

Initial direction

What was tried initially?
After finishing the prompt hardening slice, the session moved “in order” to broader API verification using full API tests and typecheck.

Why didn’t it work?
The typecheck failures were unrelated to the prompt slice. They came from pre-existing strict typing mismatches in route test fixtures (annotations.test.ts, evaluators.test.ts, scores.test.ts).

Evidence: > pnpm --filter @foxhound/api typecheck src/routes/annotations.test.ts(...): error TS2345 ... ERR_PNPM_RECURSIVE_RUN_FIRST_FAIL @foxhound/api@0.0.1 typecheck

This was not a bad verification instinct, but the repo lacked a declared baseline of known typecheck debt. That made the failure look like new breakage until investigated.

Correct approach

What was the correct approach?
Inspect the failing fixture types directly and patch them narrowly: add as const literals, required fields like completedAt, and correct mocked return values.

Before: pnpm --filter @foxhound/api test && pnpm --filter @foxhound/api typecheck After: read failing test files edit fixture literals / mocked return types re-run typecheck Final evidence: > @foxhound/api@0.0.1 typecheck > tsc --noEmit

Preventive repo improvement
Document known verification debt and establish a “slice verification vs baseline repo repair” distinction.

Suggested repo changeLocationEstimated savings
Add a “Known verification debt” section listing existing failing suites or strict-type fixture hotspots. docs/reference/engineering-notes.md or docs/plans/testing/ 5–8 minutes per verification pass
Add typed test-fixture helpers for common entities instead of ad hoc object literals. apps/api/src/test/fixtures/ 5–10 minutes per future type-tightening change
Note in AGENTS/CLAUDE: when broad verification fails outside the touched slice, classify as baseline debt before assuming regression. CLAUDE.md 2–4 minutes of investigation churn
Friction Point 3

Used an edit anchor that was too generic for exact-match editing

Initial direction

What was tried initially?
The session attempted to patch scores.test.ts with an edit block whose oldText occurred multiple times.

Why didn’t it work?
The edit tool requires each oldText block to be unique in the original file. The selected anchor matched two places.

Tool failure: Found 2 occurrences of edits[0] in .../scores.test.ts. Each oldText must be unique. Please provide more context to make it unique.
Correct approach

What was the correct approach?
Re-anchor the edit with surrounding test-name context so each replacement targets exactly one block.

Before: oldText = repeated db.getTrace mock block After: oldText = block including surrounding test case: it("creates score when valid", async () => { mockApiKey("org_1"); vi.mocked(db.getTrace)... Result: Successfully replaced 5 block(s) in scores.test.ts.

Preventive repo improvement
Codify repo-local editing guidance for exact-text tools: use nearest test name or function signature as anchor.

Suggested repo changeLocationEstimated savings
Add a small “Editing conventions for agent tools” section: anchor replacements with nearest describe/it or function signature. CLAUDE.md or repo AGENTS.md 1–3 minutes per failed patch
