Foxhound is fundamentally set up around the right core: a TypeScript monorepo, Fastify API, Drizzle/Postgres, BullMQ worker, typed clients/SDKs, and a clear multi-tenant security posture. The problems are less “wrong foundation” and more “too many surfaces grew at once, faster than consolidation.”

Architecture in plain English

  • Apps: apps/api, apps/worker, apps/web
  • Core runtime: Fastify API + BullMQ worker + Postgres/Neon + Redis/Upstash
  • Shared packages: DB, API client, SDK, Python SDK, CLI, MCP server, billing, notifications, types, demo-domain
  • Docs/public surfaces: internal docs/, public docs-site/, GitHub Action quality gate
  • Design intent: one codebase serving product UI, ingestion, async compute, and external developer surfaces

System shape

    ↘            ↙

This is the right topology for a small-team observability product that needs both control-plane correctness and async analytics workflows.

What it got right

1. Correct service split

Separating API request handling from worker jobs was the right early decision. Evaluators, experiments, cost monitors, SLA checks, and regression detection belong off the request path.

2. Multi-tenant discipline is explicit

The repo repeatedly encodes the org_id rule in docs and code. That’s exactly what a security-sensitive infra product should do.

3. Drizzle + Postgres was the right starting store

For the current product stage, Postgres is an excellent system of record for auth, config, billing, prompts, evaluations, datasets, and even early telemetry.

4. External surfaces are first-class

Typed API client, SDKs, CLI, MCP server, docs site, and CI action all show good platform instincts. This is not “just an internal app.”

5. Thoughtful telemetry modeling

The hybrid trace model—summary record plus normalized spans, with JSONB where flexibility helps—is pragmatic, not dogmatic.

6. Docs know the architecture’s real scaling limit

The architecture docs correctly identify that Postgres is fine now but not the forever analytics engine. That’s unusually clear-headed.

Best architectural decision in the repo

Treating Foxhound as a control plane + telemetry platform, not just a dashboard. The API, worker, SDK, MCP, CLI, and docs all reinforce that. That gives the product leverage.

What it got wrong or only half-right

1. Too many product surfaces at once

The repo carries API, worker, web dashboard, TS SDK, Python SDK, CLI, MCP server, docs-site, GitHub Action, and demo-domain simultaneously. Ambitious, but expensive in maintenance and coherence.

2. Monoliths re-emerged inside the monorepo

packages/db/src/queries.ts is still a high-risk gravity well even after partial decomposition. Similar issue in large API/client/MCP entry files.

3. Client logic is duplicated across packages

The API client and SDK both implement raw fetch layers and overlapping endpoint wrappers. That increases drift risk and slows interface cleanup.

4. Web layer is strategically ahead of backend maturity

The dashboard is broad and thoughtful, but much of it still behaves like a wireframe/product shell. That’s fine, but it means the front-end surface area outpaced backend depth.

5. Process artifacts are bleeding into repo truth

There are many active plans, review HTMLs, session decks, and generated audit artifacts. Some are useful; together they create retrieval noise and “which document matters?” overhead.

6. Telemetry persistence path is still transitional

The architecture doc says the right thing about future ClickHouse/object storage, but the codebase still reflects a Postgres-first world with micro-batch buffering. That’s acceptable now, but clearly an upcoming fault line.

Bloat and tech debt inventory

Structural debt

  • Oversized query surface in DB package
  • Large MCP server entrypoint with many tools in one file
  • Large typed client surface in one file
  • SDK namespaces hand-roll transport instead of sharing more infrastructure
  • API app entrypoint manually registers many routes one by one

Surface-area bloat

  • Too many near-simultaneous product/package bets
  • Dashboard breadth exceeds likely daily usage maturity
  • Docs-site + internal docs + session decks + plans all active
  • Demo-domain exists because product presentation needs are real, but it adds another axis to maintain

Hygiene debt

  • Local generated artifacts in app trees (.next, .turbo, node_modules)
  • Readmes that lag reality in places, especially around “in progress” vs already-shipped UI breadth
  • Too many active plans for one repo state snapshot
  • Mixed durable truth and review artifacts in nearby locations

Top 5 tech-debt items I’d fix first

  1. Finish decomposing DB query layer by domain and stop exporting giant grab-bags.
  2. Split MCP tools into per-domain modules with a tiny registration index.
  3. Unify transport/client primitives so SDK and API client share more code/contracts.
  4. Prune active plans/review artifacts into a tighter “current truth vs archive” split.
  5. Introduce a sharper control-plane vs telemetry-read boundary in code before adding more analytics features.

What I would not call debt

  • Using Postgres today
  • Using BullMQ today
  • Having a monorepo
  • Having a Python SDK and TS SDK together
  • Having an MCP server at all

Those are category-appropriate choices. The debt is mostly in how much got layered on top without enough consolidation passes.

If I were starting from scratch again

My setup, knowing what this review revealed

Phase 1: Ship the spine only

  • Monorepo still
  • Apps: API, worker
  • Packages: db, types, api-client, sdk, sdk-py
  • No CLI/MCP/docs-site/web until the core event model stabilizes

Phase 2: Add 1 operator surface

  • Add either MCP or web first, not both together
  • I’d probably add MCP before full dashboard
  • Use it to validate debugging workflows before visualizing every page

Phase 3: Add distribution layers

  • Docs-site after the public API/SDK contracts settle
  • CLI only if real user workflows justify it
  • GitHub Action after experiment/eval path is stable

Clean-slate architecture I’d choose

  • Repo: pnpm + Turborepo monorepo
  • Backend: Fastify + Zod + Drizzle
  • DB: Postgres as system of record
  • Jobs: Redis + BullMQ
  • Analytics roadmap: explicit future ClickHouse lane from day 0, but not implemented until pain justifies it
  • Package boundaries: domain-auth, domain-traces, domain-evals, domain-prompts style modules rather than giant query/util files
  • Contracts: one canonical HTTP transport/shared schema layer used by SDK, CLI, and MCP

What I’d consciously avoid

  • Building a broad dashboard before operator workflows are proven
  • Letting docs/plans/session decks all live as equally “active” truth
  • Hand-maintaining many large entry files
  • Shipping overlapping external surfaces without a shared contract core
  • Keeping telemetry analytics tightly coupled to Postgres query shapes for too long

Rewrite or refactor?

Refactor. I would not restart this repo from zero unless the goal were explicitly to reduce product scope. The current repo already made many correct choices. The win is in consolidation, not reinvention.

If I were your principal engineer, I’d recommend a 3–5 week simplification campaign: collapse large modules, reduce documentation noise, tighten package contracts, and formalize the future telemetry split—without changing product direction.

Bottom line

Foxhound is not an overengineered mess. It is a promising, strategically well-aimed platform that has accumulated the normal debt of a team building product, platform, and distribution at the same time. The answer is not a ground-up rewrite. The answer is to preserve the core architecture and run a hard consolidation pass.
