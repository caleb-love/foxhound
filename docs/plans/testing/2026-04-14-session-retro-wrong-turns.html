#1 · Package resolution drift

Tried to fix runtime by rewriting source imports instead of fixing workspace install state

  • Initial try: Changed packages/demo-domain/src/index.ts from explicit .js imports to extensionless TS-friendly imports so Next dev would stop 500ing.
  • Why it failed: It fixed runtime temporarily but broke package typecheck because demo-domain is NodeNext-configured and expects explicit .js import specifiers in TS source.
  • Correct approach: Restore NodeNext-correct imports, then discover the real issue: @foxhound/demo-domain was declared in apps/web but not actually linked into apps/web/node_modules until pnpm install was run.
  • Preventive repo improvement: Add a short repo note in apps/web/README.md and/or CLAUDE.md: “If a workspace package fails to resolve in web dev, verify it is actually linked in apps/web/node_modules before editing TS import specifiers.”
BeforeWrong approach
Root cause assumed: TS/Next import extension mismatch
Action taken: rewrite demo-domain src exports away from .js imports
Side effect: package typecheck failure under NodeNext
AfterCorrect approach
Evidence found: apps/web/node_modules/@foxhound lacked demo-domain symlink
Action taken: pnpm install
Result: package linked, runtime + typecheck path reconciled
#2 · TS alias detour

Tried to solve package resolution with TS path hacks that fought runtime

  • Initial try: Removed and then reintroduced TS path mappings in apps/web/tsconfig.json, including pointing @foxhound/demo-domain at generated declarations.
  • Why it failed: The TypeScript resolver and Next runtime were pulling in different truths. The path trick helped typecheck but regressed module resolution in the running dev server.
  • Correct approach: Treat package install/link state as the real source of truth, fix that first, then keep TS config simple.
  • Preventive repo improvement: Add a “workspace resolution debugging order” note in CLAUDE.md / docs/overview/start-here.md: 1) verify package symlink, 2) verify package exports, 3) only then touch TS paths.
BeforeWrong approach
apps/web/tsconfig.json temporarily used @foxhound/demo-domain -> dist/index.d.ts
Typecheck improved, runtime regressed
AfterCorrect approach
apps/web/node_modules/@foxhound/demo-domain symlink restored
TS path workaround removed
Runtime and typecheck both green
#3 · Brand truth ambiguity

Assumed old docs-site orange tokens were the brand source of truth

  • Initial try: Inferred default theme from local docs-site CSS that still used orange tokens.
  • Why it failed: That source was stale. The real current marketing implementation lived in a different repo (foxhound-web) and used a blue-primary system.
  • Correct approach: Read the actual marketing repo’s design system file, then align the default tenant theme to that source of truth.
  • Preventive repo improvement: Add an explicit “brand source of truth” section to CLAUDE.md or a new docs/reference/brand-source-of-truth.md naming the authoritative repo/file for marketing tokens.

#4 · Demo route completeness discovered late

Tried to avoid broken routes by removing links instead of first checking for reusable existing components

  • Initial try: Removed or de-emphasized broken demo links like replay/session paths after seeing 404s.
  • Why it failed: It reduced brokenness short-term but didn’t solve the product problem, and it hid that reusable replay/evaluator components already existed in the codebase.
  • Correct approach: Search the repo for existing dashboard implementations and wire demo routes on top of shared components.
  • Preventive repo improvement: Add a route inventory or information architecture doc for demo mode, listing implemented routes, missing routes, and shared components available to backfill them.
BeforeShort-term mitigation
Replay/session links suppressed or redirected away from missing pages
AfterActual fix
Found existing replay/evaluator components in apps/web
Added /demo/replay, /demo/replay/[id], /demo/evaluators, /demo/sessions/[id]
#5 · Repeated exact-match patch friction

Large anchored replacements kept failing on duplicated UI patterns

  • Initial try: Patch repeated JSX blocks with large exact snippets across deeply similar cards and records.
  • Why it failed: Many files contained repeated structures (cards, badges, sections), so generic matches collided or missed after nearby edits.
  • Correct approach: Re-read the file and patch smaller unique chunks using surrounding labels/titles as anchors.
  • Preventive repo improvement: Add a short repo rule under exact-text editing guidance: “For repeated UI structures, anchor edits with the nearest unique title/label, not repeated class blocks.” Also consider more helper extraction to reduce repeated markup.
BeforeWrong approach
Large exact block patch against repeated card structures
Result: duplicate-match failures and re-read loops
AfterCorrect approach
Patch each repeated block with unique nearby content (e.g. specific CardTitle or section heading)
Result: reliable exact-match edits

Executive takeaway

  • The session was productive, but a meaningful share of time went to truth discovery, not implementation.
  • The two most expensive wrong turns were both rooted in missing source-of-truth documentation.
  • Adding a small number of repo-native docs would likely save 45–80 minutes in similar future sessions.
