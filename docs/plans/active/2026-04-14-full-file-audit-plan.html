Highest-leverage debt themes
The repo’s biggest backlog is not missing implementation breadth. It is truth hygiene: generated artifacts and caches committed into the working tree, weak verification floors in shared security-critical layers, packaging/release inconsistency, and a docs/process surface that has become much larger than the enforcement surface underneath it.

Top findings

Critical

Generated artifact sprawl

1,174 in-scope files are generated/cache/noise. This includes tracked dist, coverage, .turbo, docs-site build outputs, docusaurus caches, tsbuildinfo, DS_Store, pycache, and Python coverage leftovers.

Critical

Secret-like files in package path

packages/mcp-server/.mcpregistry_github_token and packages/mcp-server/.mcpregistry_registry_token should not live inside a package directory, even if gitignored.

Critical

Shared-layer test debt

packages/db remains the biggest hardening gap. The tenant-safety core is concentrated there while coverage is low and implementation is monolithic.

High

Docs/process bloat

The repo has a very large planning/session/archive surface. Useful, but currently bigger than the enforcement surface in CI. There is risk of “documentation feels hardened” while runtime verification lags.

High

Public package inconsistency

SDK, MCP, CLI, and Python package release surfaces are not equally hardened. CLI still lacks test scripts. SDK threshold policy is permissive. Python package has local caches committed.

High

Authored/generated truth blur

Example: packages/types/src/index.js, index.d.ts, and maps in src/. Generated outputs are mixed into authored source directories.

Execution phases

Phase A — Sanitation

Remove tracked noise, token files, cache/build artifacts, and restore source-of-truth clarity.

Phase B — Enforcement

Add CI gates for artifact cleanliness, secret surfaces, changed-package verification, and coverage floors.

Phase C — DB hardening

Split queries.ts, deepen tenant-safety tests, and raise core verification confidence.

Phase D — API/worker resilience

Expand route abuse-path and worker operational-path verification.

Phase E — Public package hardening

CLI/MCP/SDK/Python packaging, tests, smoke installs, release simulation.

Phase F — Docs/process pruning

Reduce durable-doc sprawl, strengthen active-truth boundaries, keep history without letting it dominate audits.
