Highest-risk debt today is not missing features. It is repo hygiene and verification debt: committed artifacts and caches, secret-bearing files under a package path, weak coverage in core shared layers, and drift between “security-critical” repo claims and the actual enforcement/verification floor.

Evidence-backed findings

Critical

Committed artifact and cache bloat

The repo contains a large amount of committed generated output and local noise outside web: dist/, coverage/, .turbo/, *.tsbuildinfo, and multiple .DS_Store files across apps/packages/docs.

Impact: bloated diffs, harder reviews, stale package surfaces, false repo truth, and CI/local drift.

Critical

Secret-bearing files exist under package path

Two sensitive token files are present under packages/mcp-server/: .mcpregistry_github_token and .mcpregistry_registry_token.

Impact: accidental exposure risk, confusing local-vs-repo boundary, and release/process fragility.

Critical

Low verification floor in core shared packages

Coverage is materially low in high-leverage layers: packages/db ~25.87% lines / 4.68% functions, packages/sdk ~58.13% lines, and the SDK config intentionally codifies a low threshold.

Impact: cross-cutting regressions can land in the repo’s most reused code with limited signal.

Concrete repo hygiene evidence

  • Root .gitignore ignores dist/, coverage/, .turbo/, and *.tsbuildinfo — yet many such files are present in tracked paths, meaning history has already absorbed them.
  • docs/ is globally ignored in .gitignore, but docs are clearly present in the repo. This indicates ignore rules are no longer truthful repo guidance and create onboarding/repo-health confusion.
  • Generated JS and declaration files exist in source trees like packages/types/src/index.js and packages/types/src/index.d.ts, blurring authored vs generated truth.
  • .DS_Store exists in root, docs, apps, packages, and dotfolders.

Security/process drift evidence

  • SECURITY.md states strict invariants, but coverage and verification for packages/db are far below what that posture implies.
  • apps/api/src/index.ts registers many security-sensitive routes in one bootstrap file, but there is no visible route-classification verification matrix in CI beyond broad lint/test/typecheck.
  • apps/worker/src/queues/evaluator.ts still contains a TODO for DLQ notifications, implying operational incident visibility debt.
  • .github/actions/quality-gate/run.ts still uses legacy fallback ::set-output behavior if GITHUB_OUTPUT is absent. Not fatal, but indicates compatibility debt and possible bundling/runtime drift.

Build/release/developer-experience debt

  • packages/cli/package.json has no test or coverage scripts, despite being a public CLI surface.
  • Multiple public package versions appear out of sync in maturity: SDK and MCP server are at 0.3.0, CLI and Python SDK are 0.1.0, with no explicit release hardening matrix in the sampled files.
  • Root CI runs broad monorepo steps, but lacks targeted gates like changed-package selective verification, package publish dry-runs, artifact cleanliness checks, secret-surface checks, or minimum coverage enforcement by risk tier.

Execution plan: exhaustible work order

Phase 0 — Repo sanitation and trust reset

  • Delete tracked build outputs, caches, coverage files, tsbuildinfo files, and platform noise outside intentionally vendored assets.
  • Remove package-local token files and replace with documented credential bootstrap outside repo paths.
  • Fix .gitignore and contributor docs so repo truth matches actual desired behavior.

Exit criteria: clean git status after fresh build/test cycle except intended sources; no secret-bearing local token files in package directories; no stray .DS_Store tracked.

Phase 1 — Verification floor hardening

  • Define risk-tier coverage policy: data/security core, public SDKs/CLI, internal packages, workers.
  • Add coverage thresholds package-by-package, replacing permissive floors like packages/sdk/vitest.config.ts.
  • Add missing test scripts to public surfaces such as packages/cli.

Exit criteria: every public package has test + coverage scripts; CI enforces explicit thresholds; no package is “unowned” from a verification perspective.

Phase 2 — Data-layer and tenant-isolation assurance

  • Break packages/db/src/queries.ts into domain modules: auth, billing, notifications, prompts, traces, evaluations, admin, etc.
  • Add exhaustive tests for org-scoped SELECT/UPDATE/DELETE behavior, cross-tenant rejection, prompt label mutation safety, API key lifecycle, and billing joins.
  • Make tenant-scope audit a required verification lane for DB changes.

Exit criteria: DB package reaches materially higher line/function coverage and has explicit tests for every sensitive query family.

Phase 3 — API and worker resilience hardening

  • Add route verification matrix: auth mode, rate limiting, tenant scope, entitlement gating, bad-input handling, and side-effect auditability.
  • Add worker tests for retries, repeated scheduler startup, shutdown semantics, DLQ behavior, and notification paths.
  • Resolve TODOs like evaluator DLQ notification wiring or formally defer them with tracked tickets.

Exit criteria: failure and abuse paths are tested, not just happy paths.

Phase 4 — Public package release hardening

  • SDK: increase tests around retry behavior, integrations, env handling, serialization, and docs examples.
  • CLI: add command tests, snapshot/structured-output tests, auth/config path tests, and install smoke tests.
  • MCP server: deepen branch/error tests and validate packaged runtime behavior.
  • Python SDK: add wheel/sdist smoke checks, optional dependency matrix, and example-based verification.

Exit criteria: all public artifacts can be built, packed, smoke-installed, and exercised in CI.

Phase 5 — Performance, operational, and doc closure

  • Set baseline performance checks for API critical paths and worker throughput.
  • Add CI checks for artifact cleanliness, package publish dry-runs, and secret-surface scanning.
  • Update durable docs: current status, engineering notes, hardening runbooks, release policy, verification matrix.

Exit criteria: the repo can stay clean and hardened without relying on memory or heroics.
