Specific friction points

Each item below captures a moment where the session went in the wrong direction first, then corrected course. The goal is not blame. The goal is to convert recurring friction into durable repo improvements.

Friction 1

Versioning policy was initially placed under ignored docs

Initial move

  • Create docs/VERSIONING.md
  • Link from docs/README.md

Correct move

  • Discover root .gitignore ignores docs/
  • Move policy to tracked root VERSIONING.md
  • Link from root README.md
Evidence - git add docs/VERSIONING.md failed because docs/ was ignored - Root .gitignore contained: docs/ - Final fix: VERSIONING.md created at repo root instead
Friction 2

Local-only repo files were treated as public content until later cleanup

Initial move

  • Proceed with repo cleanup while AGENTS.md and CLAUDE.md were still present in working tree/public path
  • Only later learn they were local Pi / local machine files

Correct move

  • Ignore all AGENTS.md and CLAUDE.md files
  • Untrack existing tracked local-only files with git rm --cached
Evidence - Added ignore rules for AGENTS.md and CLAUDE.md late in session - Needed git rm --cached for tracked CLAUDE.md files because ignore rules do not affect tracked files - This was discovered only after status still showed modified CLAUDE.md
Friction 3

Package publishing was attempted before package surfaces were release-clean

Initial move

  • Attempt npm publish for SDK and MCP packages
  • Assume auth was the only blocker

Correct move

  • Inspect npm pack --dry-run output first
  • Clean SDK and MCP package contents
  • Add .npmignore and files allowlist
Evidence Before cleanup, MCP dry-run tarball contained: - .mcpregistry_github_token - .mcpregistry_registry_token - .turbo logs - coverage/ - mcp-publisher - test sources and test build artifacts After cleanup: - MCP tarball reduced to 9 files - SDK tarball no longer included compiled test artifacts
Friction 4

CI diagnosis initially focused on lockfile/install assumptions instead of the actual failing step

Initial move

  • Suspect pnpm install --frozen-lockfile mismatch
  • Refresh local install and inspect lockfile

Correct move

  • Run pnpm format:check locally
  • Confirm formatting step was failing in CI
  • Run Prettier and push style-only fix
Evidence - Local pnpm install reported lockfile up to date - Local pnpm format:check failed on 82 files - After formatting pass and final file cleanup, format:check passed - Commit pushed: style: run prettier across repo for CI
Friction 5

Python Hatch fix was applied to the wrong env key first

Initial move

  • Add missing Python test deps under [tool.hatch.envs.hatch-test]
  • Then switch to [tool.hatch.envs.test]

Correct move

  • Use CI log evidence showing environment name hatch-test.py3.11
  • Mirror dependencies into both test and hatch-test env configs
Evidence Repeated CI failure: - ModuleNotFoundError: No module named 'langchain_core' - Unknown config option: asyncio_mode - Unknown pytest.mark.asyncio This proved langchain-core and pytest-asyncio were still not installed in the actual Hatch CI env.
Friction 6

Local GitHub action referenced build output that was not committed

Initial move

  • Treat action failure as runtime/module issue
  • Only later inspect local action bundle tracking state

Correct move

  • Check whether .github/actions/quality-gate/dist/run.js was tracked
  • Commit only the required bundle
  • Ignore local action .turbo/ junk
Evidence Action error: - Cannot find module .../.github/actions/quality-gate/dist/run.js Repo inspection showed: - dist/run.js existed locally - git ls-files did not include it Fix applied: - commit .github/actions/quality-gate/dist/run.js - ignore .turbo/ in action directory

Before / after evidence

The pattern was consistent: the wrong direction usually came from repo truth being implicit instead of explicit. Once the actual truth was read from files or command output, the fix was straightforward.

Before: ambiguous repo truth

Examples - docs/ treated as durable truth in repo guidance, but root .gitignore ignored docs/ - local-only AGENTS.md and CLAUDE.md were present without repo-level public/private boundary guidance - local action expected dist/run.js but tracked files did not include it - Python test env behavior was assumed, not codified clearly - formatting policy existed in CI but repo contained many unformatted files

After: explicit correction

Examples - VERSIONING.md moved to tracked root - AGENTS.md and CLAUDE.md ignored / untracked as local machine files - SDK and MCP npm surfaces cleaned via .npmignore and files allowlists - quality-gate bundle committed intentionally - formatting normalized and CI-specific blockers fixed with dedicated commits

Ranked repo improvements

Rank 1

Fix durable-truth location policy

  • Either stop ignoring docs/, or stop telling agents durable truth lives there.
  • Make one documented public location for versioning, plans, and repo process truth.
Estimated savings: 10-15 min per substantial docs/process change
Rank 2

Codify local-only file boundaries

  • Document that AGENTS.md and CLAUDE.md are machine-local in this repo.
  • Keep ignore rules in root by default.
Estimated savings: 8-12 min per repo hygiene pass
Rank 3

Require npm pack dry-runs before publish

  • Add a documented release checklist.
  • Optionally add a script that fails if package tarballs include test or token files.
Estimated savings: 12-20 min per release, plus major risk reduction
Rank 4

Add CI diagnosis guide to repo ops docs

  • Document the workflow order: install, lint, format:check, typecheck, test, Python test, build.
  • Advise reproducing exact failing steps locally before theorizing.
Estimated savings: 6-10 min per CI incident
Rank 5

Document Hatch test env contract

  • State which env key is used in CI.
  • List integration-test dependencies explicitly.
Estimated savings: 10-15 min per Python CI fix
Rank 6

Define local-action bundling rule

  • If a local GitHub Action references built JS, the built artifact must be committed.
  • Ignore local build logs and caches in the action directory.
Estimated savings: 8-12 min per action-related CI issue
